# Usage/quota introspection (Step 3 adapter)

Findings from a concrete check (2026-07-10) ‚Äî re-verify before assuming anything here has changed in a newer codex version.

**Update (2026-07-11, lived):** Claude Code's `/usage` also fails via `--print` (echoed back as literal text), **but ‚Äî unlike codex ‚Äî it works non-interactively via stdin into an interactive session**: `echo "/usage" | CLAUDE_CONFIG_DIR="<profile-dir>" claude 2>&1` (no `--print`; `/exit`/`sleep` unneeded ‚Äî it exits on EOF in ~2 s), then parse `Current session: NN% used`. It returns the true subscription percentages, not a proxy ‚Äî **but it is best-effort**: those percentage lines intermittently fail to render on rapid repeated calls (the "What's contributing‚Ä¶" section always does), so query at most once per cycle and treat a miss as no-data, not 0%. Codex, by contrast, has an even cleaner *structured* channel ‚Äî a JSON-RPC endpoint, not a TUI scrape ‚Äî documented in the 2026-07-12 finding below. The Claude proxy remains **`ccusage`** (token burn from `~/.claude/projects/*.jsonl`, an approximate active 5-hour block ‚Äî *not* the subscription %).

**Codex ‚Äî the working non-interactive channel (2026-07-12, lived and preferred; supersedes the 2026-07-10 "no official check" conclusion).** Do **not** use `/status`: piped through `codex exec` it becomes a literal prompt (the agent starts working), and piped into the interactive TUI it errors `stdin is not a terminal` (the TUI needs a real TTY). Instead drive `codex app-server --stdio` over **JSON-RPC** ‚Äî codex's *official (experimental)* app-server protocol, returning **structured JSON**, cleaner than any status-screen scrape and cleaner than Claude Code's `/usage`:

- **Transport:** newline-delimited JSON (one JSON object per line) on stdin/stdout ‚Äî **not** LSP-style `Content-Length` framing (that fails to deserialize).
- **Handshake:** send `initialize` (params `clientInfo` + `capabilities`), await the `id`-matched result; send the `initialized` notification; wait ~500 ms; send `{"jsonrpc":"2.0","id":2,"method":"account/rateLimits/read"}`; read the `id:2` result. Ignore interleaved notifications (e.g. `remoteControl/status/changed`). (An `account/usage/read` method also exists.) Discover the exact schema anytime with `codex app-server generate-json-schema --experimental`.
- **Fields to parse** under `result.rateLimits`: `primary.usedPercent` + `primary.windowDurationMins` (**300 = the 5-hour window**) + `primary.resetsAt` (unix seconds); `secondary.usedPercent`/`windowDurationMins` (**10080 = weekly**)/`resetsAt`; `planType`; `credits.{hasCredits,unlimited,balance}`; and `result.rateLimitResetCredits.availableCount`. **`usedPercent` is % USED** (the TUI shows % *left*) ‚Äî so hand off when `primary.usedPercent` ‚â• your 95 threshold.
- **Reliability (lived):** ~1.1‚Äì1.2 s typical, but use a **hard timeout ‚â• 15 s** and **retry** ‚Äî one call in five returned `503 Service Unavailable` from the ChatGPT backend, and the snapshot can *flap* between two values on rapid fresh calls (server-side eventual consistency). Treat a transient failure/flap as "no reading this cycle" (non-blocking, per Step 3); if a decision is close, read twice and take the higher `usedPercent` (conservative ‚Üí hand off earlier). Proven on Windows, codex-cli 0.142.5, plan `plus`.
- `codex doctor` / `codex login status` still expose nothing, and `codex exec --json`'s `turn.completed.usage` is per-call tokens only, not account quota.

**Unofficial third-party tools ‚Äî now largely redundant** given the app-server method above (which is the same interface, driveable directly with no extra install): `codex-status-json` wraps exactly `codex app-server`'s `account/rateLimits/read`; `pi-codex-status` queries ChatGPT's backend with the OAuth token. Prefer the direct app-server call; reach for these only if you want a packaged binary, and note they touch auth material / undocumented endpoints.
