# Anti-patterns observed (avoid repeating)

Almost every entry here was lived, not hypothesized. The narrow exception is a guard against a rare, hard-to-detect, irreversible catastrophic risk (secret leak, unauthorized deploy, simultaneous orchestrators, data loss), admitted pre-incident under SKILL.md's "Keeping this skill alive" rule and marked *(pre-incident guard)*: the succession/handoff cluster below was worked out from a concrete overnight-continuity scenario rather than lived one incident at a time. Read this catalog at bootstrap and whenever a failure in progress feels familiar — the matching entry usually names the rule that prevents it.

- Assuming a tool's file-write sandbox works because it worked last session on a different machine — re-run the Step 0.3 smoke test per machine.
- Accepting "tests passed" or "verified" from a dispatched agent without independently re-running the same check.
- Waiting indefinitely on a stalled dispatch instead of applying the stall rule.
- Silently reusing account/profile names or role assignments from a previous machine instead of running fresh discovery.
- Skipping the Step 2 gate definition and only writing a prose task description — this is what produces unverifiable "looks done" outcomes.
- Trusting that every component of the project reaches the shared runtime environment the same way (all live-linked, or all requiring manual sync) instead of checking each one — this is what makes an edit silently fail to appear, or silently land somewhere nobody is testing.
- Treating a bead as "not started" or "stuck" based on its tracker status alone, without checking whether the real work already landed — committed and complete — in an isolated worktree or branch that just never got merged.
- Continuing to dispatch and verify work as the orchestrator without ever checking its own remaining usage, then running out of room mid-bead with no handoff snapshot written — leaving in-flight worktrees and unverified dispatches for a future session to rediscover from scratch.
- Starting orchestration without a declared succession plan or worker-exhaustion policy, then improvising a different response each time usage runs low — the whole point of asking once at bootstrap is to have a standing answer instead of a fresh judgment call every time, especially when nobody is around to make one.
- Treating gate-verification (Step 2) as if it were also a code review — running the lint/test/QA check and merging without a second, deliberately separate look at anything beyond a trivial change.
- Writing code directly as the orchestrator by default instead of delegating, turning "I'll just fix this myself" into a habit rather than the exception it should be for genuinely cheaper-than-dispatch fixes.
- Blocking an entire unattended run on one question nobody can answer for hours, instead of falling back to the standing bootstrap policies that exist specifically for this.
- Assuming a handoff "just works" without ever verifying, on this environment, that a spawned successor actually starts operating — treating the handoff mechanism as reliable by assumption instead of confirming it the way Step 0.3's write-smoke-test confirms file-write capability.
- Handing off permanently instead of as a temporary regency — letting a successor keep the orchestrator role indefinitely once it has it, instead of returning control at the original orchestrator's own `reclaim_at` once it's actually available again.
- Starting a new orchestrator session without checking for an existing orchestrator lock, risking two identities dispatching and merging work at the same time. *(pre-incident guard — simultaneous orchestrators.)*
- Dispatching two beads in parallel that touch the same files without noticing, then discovering the conflict only at merge time instead of serializing them upfront.
- Leaving "does this need a dedicated review" purely to in-the-moment judgment on paths the user already flagged as always-sensitive at bootstrap — a declared list exists precisely so this isn't re-litigated, or skipped, under pressure.
- Dispatching a worker without an explicit instruction not to read or output secrets/credentials, on the assumption the tool's own training already covers it.
- Treating a closed or merged bead as verified just because it's closed or merged — external merge drift (0.12) means work can land on `main` through channels that never passed this skill's own gate.
- Silently absorbing an out-of-scope finding into the current bead instead of surfacing it and letting the user decide — this is what turns "verify these 3 beads" into an unplanned, unannounced audit of everything.
- Trusting a browser tab's error display without a direct HTTP cross-check — reloading the same glitched tab isn't always enough to tell a rendering fluke from a real server-side failure.
- Leaving a shared-runtime indirection repointed at a dispatched agent's own worktree after that agent's task is done, instead of making restoring it part of the task's own definition of done.
- Declaring a dashboard update "done" after re-running its auto-refresh script, without confirming on screen that a newly created bead is actually visible — a hand-curated layer the script doesn't touch can leave it silently missing.
- Retrying the same failing bead indefinitely across different tools instead of respecting the 2–3-attempt budget — three defeated dispatch cycles point at the task definition or environment, not at the choice of worker.
- Merging a worktree onto a main that has moved since the worktree was created, trusting a gate that passed against the old base — and never re-running the gate on main itself after the merge, so the state everyone actually builds on is the one state never verified.
- Grabbing the top of `bd ready` mechanically instead of choosing deliberately — starting a large bead minutes before a usage boundary, or letting a high-unblocking-power bead sit while low-value ones get dispatched.
- Dispatching two beads in parallel because their file sets are disjoint, without checking the bd dependency graph — building one on foundations the other is actively changing.
- Treating a worker's `bd close` as a verified state instead of a completion signal — the closing comment is input to verification, not a substitute for it.
- Letting a freshly-learned failure mode live only in session memory until the end of the stretch, instead of recording it via `bd remember` the moment it happens — end-of-session summaries flatten the details that make a rule worth writing.
- Installing this skill — or any shared artifact — by copy, so a stale duplicate serves some sessions while every edit lands in a different copy (lived: a `CLAUDE_CONFIG_DIR` profile ran a 13.9k-word pre-modularization monolith while all edits went to `~/.claude/skills`). This is 0.3's "edits landing in a copy nobody serves," lived by the skill file itself. On a multi-profile machine, junction/symlink each profile to one canonical copy; never install by copy.
- Silently skipping a CONDITIONAL/optional bootstrap sub-step because it is "optional" (lived **twice**: the 0.11 dashboard was never proposed and the user had to ask for it both times, even after the end-of-Step-0 accounting rule was added — a reporting requirement buried at the end of a noisy bootstrap is too weak). "Optional" means user-declinable, never orchestrator-skippable: make the proposal a concrete, upfront ask *during* Step 0 ("shall I set up the dashboard from this template?"), not a line in a closing summary.
- Running an orchestration with no durable on-disk session log, so when the machine sleeps or reboots the entire reasoning trace is gone and the user has to screenshot the chat from their phone to keep any record (lived: exactly that on the first overnight run). Open the Step 0 session log before 0.1 and append to it throughout — install actions included. **Recurred a second time in a different project** despite that framing — a whole bootstrap ran, hit four real incidents, and only got a reconstructed-after-the-fact log once asked directly, the same meta-failure as the 0.11 dashboard proposal getting skipped twice before its rule was hardened from prose into a structural marker. The session-log instruction sits as bold prose *before* the numbered 0.1–0.12 list rather than carrying the same `— BLOCKING` tag those items scan for; treat it as needing the same visual weight, not just the same wording, so it doesn't read as optional filler ahead of the "real" checklist.
- Keeping a session log but letting it lag and batch-catching-it-up only when the user asks — the un-flushed span (last update mid-round, dozens of lines behind, through a bead failure and a whole investigation) is exactly what a crash or shutdown destroys, which is the loss the log exists to prevent (lived). Append at each bead-cycle boundary (Step 1.12) and immediately on each incident, like the snapshot and the ledger — a "keep it current" phrased as a background intention gets deferred, so tie it to the concrete per-cycle checkpoint. **Honest status: tying it to Step 1.12 has not actually stopped the recurrence** — the same session that self-reported this lesson later lagged behind *again*, twice more, in that same session, only corrected when the user explicitly called it out mid-session a second time ("write in the log that you're not updating it often") (lived). Do not present the Step 1.12 fix as solved; a per-cycle checkpoint that competes with "attention goes to verifying/merging the next bead first" during long sessions with frequent parallel dispatch is a necessary but evidently insufficient mitigation — treat every long, high-dispatch-density session as still at risk for this and check log recency explicitly, not just at cycle boundaries.
- Dispatching real background workers before verifying the headless launch path is safe (lived, all in one run: Codex `sandbox="elevated"` fired a UAC prompt per command — dozens; workers hung forever because stdin never got an EOF in background; a stray `&` inside an already-backgrounded launch reported a false early completion; an `--allowedTools` flag swallowed following prompt text). Launch-smoke-test the launch path in 0.2 the way 0.3 smoke-tests file writes.
- Detecting completion/stall/error with a grep over the whole growing worker log instead of only the new tail — it re-matches historical occurrences and, worse, matches your own dispatch instructions (`bd close <id>`) echoed back into the log, producing a false "done" (lived).
- Trusting a red gate produced inside a worker's own sandbox as a real failure when it was a sandbox limitation — e.g. `EPERM` blocking the subprocess forking vitest's `pool:forks` needs (lived). Re-run every gate yourself outside any worker sandbox before believing red.
- Scoping a dispatch from `bd show` when it displayed `(none)`, while the bead's full detailed spec actually existed in the raw `issues.jsonl`/Dolt source (lived: a 4-phase, 879-occurrence token migration got dispatched as something far narrower). Read the ground-truth record, and retro-check prior decisions when you find such a display bug.
- Letting each worktree run its own install of gitignored build deps (`node_modules`) instead of seeding them from main — native postinstall (esbuild) and Windows `npm ci` cache-lock `EPERM` send the worker down a self-diagnosis rabbit hole instead of its actual gate (lived). Copy main's already-good deps into the worktree.
- Merging a worker's sandbox-workaround scratch/debug files (e.g. a stray `vitest.node.config.*`) as if they were part of the feature (lived). Prohibit scratch files in the dispatch prompt and remove any before merge.
- Reading a worker's diff that shows *every line changed* as real churn when it is just line-ending (CRLF/LF) normalization display noise — confirm against `git diff` on the merged result, which normalizes, not the worker tool's own diff view (lived).
- Reading `/usage` (or codex `/status`) via `--print` and concluding there is no usage channel — `--print` echoes the command as literal text, so the handoff trigger silently never fires and the gap surfaces only when someone asks why (lived). For Claude Code the working channel is `/usage` piped as stdin into an interactive `claude` session (`echo /usage | CLAUDE_CONFIG_DIR=<dir> claude`, no `--print`, no `/exit`), polled-for and parsed — but **best-effort**: the percentage lines intermittently don't render on rapid repeats, so query once per cycle and treat a miss as *no data this cycle*, not 0% and not a stop. For codex the working channel is the *structured* `codex app-server --stdio` JSON-RPC `account/rateLimits/read` (not `codex exec`, which echoes) — cleaner than the Claude scrape, though it too can 503 or flap and needs a timeout + retry. Only a truly headless env (no launchable interactive session) makes it unavailable — then fall back *non-blockingly*, never a check-in no one is awake to answer.
- Deciding a worker or subprocess has ended by `pgrep`/pattern-matching its name — it can report ENDED while the process is still alive (lived, repeatedly, with the launcher wrapper). Verify termination by the **exact PID** (`ps -p <pid>` / the process-group check from the 0.2 launch manifest), never a name pattern.
- Believing a worker's account of which files it wrote — under sandbox write-denials it can be self-contradictory (lived: it claimed one directory both succeeded and was denied, and wrote only 3 of ~9 files). Establish the truth from `git status`/`git diff` in the worktree, clean up leftover probe/zero-byte/CRLF-only artifacts, and be ready to finish the work yourself. (Windows write-denial incident, previously open with root cause unidentified: see the resolved Codex `apply_patch` entry below — the fix there (worktree under the home directory) is the confirmed mitigation; `--sandbox danger-full-access` is no longer needed and should not be reached for without the user's explicit sign-off regardless.)
- Dispatching Codex on Windows into a worktree created as a sibling of the main repo (e.g. `../repo-wt-<id>-a<n>`, outside the user's home directory) — its `apply_patch` refuses every write with "windows unelevated restricted-token sandbox cannot enforce split writable root sets directly; refusing to run unsandboxed," confirmed **systemic across three separate attempts in one session**, with the worker's own fallback then hanging silently instead of failing fast. Root cause and fix confirmed later: the sandbox can only compute a writable-root set when the worktree sits under the home directory. Fix: on Windows, always create a Codex worktree under the home directory (e.g. `~/codex-worktrees/<repo>-<id>-a<n>`), never as a sibling of the main repo — confirmed by a clean first-attempt success immediately after applying it. This does not relax the Step 1.3 sibling-path rule for the repo layout itself; it adds a Windows/Codex-specific placement constraint on top of it. **The home-dir fix resolves that specific sandbox-refusal signature, not every `apply_patch` failure** — a follow-up session hit one plain, generic "patch: failed" (no sandbox-refusal text) writing a single new file inside an already-home-dir worktree; Codex self-recovered by falling back to a native shell write (PowerShell `Set-Content`) for that one file. Treat a differently-worded `apply_patch` failure as an ordinary retryable tool hiccup — not evidence the home-dir fix didn't hold — and a worker's own fallback to native file-write tooling as an acceptable recovery to notice, not a red flag on its own.
- Overriding a reviewer's blocking ("not safe to merge") verdict with the orchestrator's own countervailing judgment instead of genuinely fixing the finding + re-review or getting explicit user sign-off (lived, established twice as precedent in one project — the classifier correctly blocked both attempts). A review exists to be a check the orchestrator is not the sole judge of.
- Accepting a worker's edit to a file outside its assigned scope because it looks helpful or harmless — a worker may edit an out-of-scope file to route around a limitation in its *own* sandbox/tooling (lived: edited `vite.config.ts` to dodge its own `spawn EPERM`). Revert it regardless; it is not the worker's call.
- Launching a dispatched worker with an elevated permission mode (`--permission-mode acceptEdits`, `--dangerously-skip-permissions`/`bypassPermissions`) because the run is unattended — the AFK authorization covers proceeding with orchestration, not granting a worker blanket edit-acceptance, and the safety classifier rightly blocks it (lived). Rely on worktree isolation + gate + independent verification, not a permission bypass; do small, fully-specified fixes yourself.
- Using a **blocking** check-in ("shall I continue?") as the *unattended* fallback for a missing usage channel — with no human watching, the run halts indefinitely on the first check-in, the exact opposite of unattended (lived corollary). A check-in cadence is attended-only; unattended, keep the snapshot fresh after every bead and schedule a usage-independent successor so reaching the rate limit is recoverable, not a question.
- Treating a tool call blocked by a transient safety-classifier/infra unavailability ("… temporarily unavailable, so auto mode cannot determine the safety of Bash") as a stall or a token-exhaustion stop — it is retryable and unrelated to budget (lived). Retry a few seconds later before declaring anything stuck or handing off.
- Trusting a helper tool's own worktree-creation default (e.g. `bd worktree create <bare-name>`) instead of an explicit sibling path — a bare name can nest the new worktree *inside* the repo root, breaking TypeScript/build tooling on duplicate package copies (lived). Always use/verify the explicit `../repo-wt-<id>-a<n>` sibling path.
- Concluding a local/experimental worker merely "got it wrong" when it actually never invoked its tools at all — printing literal tool-call syntax as assistant text instead of the harness executing it is a deeper failure than wrong content, and can happen even on short simple tasks (lived, more severe than that profile's prior "fails only on complex prompts" record). Confirm with `git status` that anything was actually written before evaluating what.
- Passing a `bd` field containing backticks/template literals/JSX as an inline shell argument (`-d "..."`) — shell interpolation can silently mangle it (lived: verified via `bd show --json` that one of three such writes was genuinely corrupted). Use `--body-file` for any description containing that kind of content.
- Considering a killed worker's worktree removable right after the kill — a killed background process can still hold the worktree directory open (Windows), so `git worktree remove` fails with a permission error until the background task is explicitly stopped, not merely abandoned (lived).
- Applying a batch of pending database migrations in filename/alphabetical order instead of real dependency order — a function defined in a later-named file but consumed by an earlier one will fail partway through (lived). Grep for definitions vs. consumers first, apply one at a time, verify each.
- Running an unfamiliar project helper script that does its own branch-switch-then-cleanup (temp branch → commit/push → `git checkout -f $originalBranch` in a `finally`) without checking `git status` on the *current* branch first — that kind of `checkout -f` discards uncommitted work sitting on the original branch at launch time, not just on the throwaway temp branch it created (lived: an uncommitted `.gitignore` edit was silently lost this way, had to be redone). Read what a project script does to the working tree before running it, the same way 0.3 already treats every repository-mutating action as needing a clean-baseline check first.
- Treating "the user asked me to delegate this task to tool X" as authorization for X's own approval-bypass/full-auto flag — a direct, explicit request to hand a task to a named tool is not the same as authorizing that tool's `--full-auto`/`--dangerously-skip-permissions`-equivalent escalation; the safety classifier correctly blocked exactly this (lived) even in an **attended** session with a clear delegation instruction, because the flag itself was never named. This generalizes the existing unattended/AFK version of the same rule (0.5): ask for the permission-bypass flag specifically, every time, regardless of how directly the delegation itself was authorized. The same non-inheritance holds **across a model swap on the same tool**: a full-auto/danger-sandbox authorization already granted for `codex exec` did not carry over to `codex exec -m <new-model>` on first use of that model — the classifier blocked it until asked again (lived) — treat a new model selection as a new dispatch needing its own authorization, not a detail of an already-authorized tool.
- Assuming every dispatched worker respects worktree isolation and leaves a `bd` comment on completion, the way the primary CLI dispatch path does — an autonomous heartbeat-based worker (lived: a profile with no blocker/dependency awareness, picking up beads on its own schedule) closed a bead with zero comments and had worked **directly on the main working tree** instead of an isolated worktree, leaving real changes sitting there uncommitted. After any such worker reports "done," check `git status` on the main repo itself, not only the dedicated worktrees this skill's own dispatch path creates — the isolation guarantee this skill relies on for safe concurrent work does not hold for every worker type.
- Declaring a visual/UI parity claim verified after confirming only the one property a bug report named (e.g. background-color), when a different property of the same element can still diverge — a fixed CTA button's color matched the target exactly while its wrapper `<div>` carried the padding/min-height/box-shadow instead of the inner link, making it 60% taller than the reference (lived); the user caught it by eye after the color-only verification had already closed the bead. For any "matches the reference" claim, check the full relevant surface — computed style **and** box-model geometry (e.g. `getBoundingClientRect`) — not just the symptom that was reported.
- Treating your own earlier-in-session visual impression as still true when opening a follow-up bead off it, instead of re-measuring it with the same objective tooling used on worker output — re-checking a morning's list of "still different" items with `getComputedStyle` before dispatching found that 2 of 4 had already been resolved as a side effect of an unrelated earlier fix, and a third ("border-left" by eye) was actually a different property (a pseudo-element `::before`) once re-inspected (lived). A stale mental note is not evidence; re-verify before creating new work from it, the same way a bead's stated premise gets checked against current reality (Step 1.4).
- Treating a cache-busted fetch of the deployed source file as proof a fix is live for real visitors — an edge/server cache layer outside the deploy tool's own reach (e.g. a hosting provider's server-side page cache) can keep serving the pre-fix version indefinitely after a verified-correct origin deploy, with no cache-control signal visible from the source fetch alone (lived). Verify the actually-served URL too; if it still diverges after the origin is confirmed correct, name a cache purge as the separate likely-out-of-reach follow-up rather than doubting the fix itself.
- Continuing to trust the orchestrator's own automated measurement tooling (e.g. a headless browser MCP) as ground truth after it has stayed persistently inconsistent with the user's direct observation in their real browser across an extended investigation (lived: over an hour, multiple CSS causes tested and ruled out via `getComputedStyle`/CSSOM in the automated environment, with no root cause ever confirmed). When automated measurement and direct human observation keep disagreeing despite genuine investigation, treat the human's direct confirmation as the decisive verification signal — not a tie-breaker to note and move past — and it is legitimate to ship a pragmatic fix with an explicit "root cause not fully explained" caveat in the ledger rather than block on an elusive full diagnosis.
- Reviewing only a worker's logic/functionality and not the literal language of the copy it wrote, on a project with a non-English (or otherwise non-default) content convention — a worker defaulted to English in an Italian-convention file not once but **twice** in the same session (an FAQ answer written entirely in English; an admin-panel label same) before either merge, caught only because the orchestrator happened to read the diff's actual text (lived). No functional gate catches this. Check copy/string language against the project's established convention as a normal part of diff review on such projects, and treat a **second** independent instance as a signal to open a dedicated audit bead rather than keep fixing occurrences ad hoc.
- Trusting a project's own previously-correct dispatch playbook/pattern doc as still accurate without re-checking it against the actually-installed CLI binary — a documented `codex exec --full-auto` pattern (recorded once, reused per Step 0.3/1.5's "reuse the exact invocation verbatim") turned out to reference a flag that no longer exists on the currently-installed `codex` version at all, confirmed only by running `codex exec --help` fresh instead of trusting the doc (lived). A once-verified invocation is not permanently verified: the underlying CLI can be upgraded between sessions and silently rename or drop flags. Spot-check a recorded invocation's flags against the installed binary's own `--help` before reusing it, especially when the recording session is not obviously recent.
- Trusting a failing real-runtime QA check without first ruling out stale test-environment state — a verification run showed what looked like a regression (an un-namespaced cookie where the fix should have produced a namespaced one), which was actually a leftover artifact from an earlier test in the *same* browser session, not a real defect; clearing cookies and repeating from a clean state showed the fix was correct all along (lived). Reset the test environment (cookies/cache/localStorage/prior test data) before concluding a real-runtime check's failure reflects the code under test rather than leftover state from a previous check.
